## 三、🔴 关键 Bug：SSE 第二次 Task 阻塞

### 问题描述

- **现象**: 第一次 `POST /api/v1/tasks` 创建任务后，SSE 事件流正常推送。第二次创建任务时，SSE 流**异常阻塞**，前端收不到任何事件。
- **后端信号源状态**: 后端 AgentLoop 和 SSEGateway 正常产生事件（日志/LLM 调用均正常），信号**在某处被阻塞**，未到达客户端。
- **影响范围**: 所有连续创建 Task 的场景。用户必须刷新页面/重启服务才能发起第二个任务。
- **疑似根因** (待验证):
  1. `HttpServer.cpp` 的 SSE chunked provider 中 `SseContext` 的生命周期管理问题 — 第一个 task 的 `ctx` 可能未正确销毁，导致第二个 task 的 `ctx->cv.wait()` 永久阻塞
  2. `SSEGateway::streamTaskEvents()` 的 `SendCallback` 绑定到已失效的 lambda
  3. EventBus 订阅未在任务完成后取消，旧订阅拦截了新任务的事件

### 复现步骤

```
1. POST /api/v1/tasks  {"global_id":"g_default","workspace_id":"w_xxx","input":"创建 hello.cpp"}
2. GET /api/v1/tasks/{taskId1}/events  → ✅ 正常收到事件
3. 等待任务完成
4. POST /api/v1/tasks  {"global_id":"g_default","workspace_id":"w_xxx","input":"修改 hello.cpp"}
5. GET /api/v1/tasks/{taskId2}/events  → ❌ 阻塞，无事件
```

### 调试线索

- 查看 `SSEGateway::liveClients_` 列表 — 根据 P1-01，此列表可能永远为空（HttpServer 绕过了 SSEGateway 的注册机制）
- 查看 `EventBus` 订阅者列表 — 旧 task 的订阅者可能未取消
- 在 `SseContext::cv.wait()` 前后加日志，确认是否永久阻塞

### 相关已有问题跟踪

- **P1-01**: SSE 双通道架构未收敛 (HttpServer 绕过 SSEGateway)
- **P1-02**: SSEGateway client removal 比较逻辑不可靠

### 4.1 Workspace 系统 — 实验代码，建议重写

**当前状态**:

```cpp
// ToolSystem.cpp — 全局单一 Workspace
workspace_ = std::make_shared<Workspace>(workspacePath);
shell_ = std::make_shared<BuiltinShell>(workspace_);
```

**问题**:
- `ToolSystem` 持有**全局唯一**的 Workspace，数据库虽然有多个 Workspace 记录，但工具执行时只使用全局路径
- `WorkspaceManager` + `WorkspaceRuntime` 已定义但**未与 ToolSystem 集成** — FileTool/ShellTool/GitTool 仍然从 `ToolSystem::getInstance().workspace()` 读取路径，而不是从 `ToolContext.workspaceRuntime` 读取
- 多任务并发时，Task B 切换全局路径会导致 Task A 的文件操作跑错目录

**建议**:
- ✅ **方案 A (推荐)**: 完成 `WorkspaceManager → WorkspaceRuntime` 集成，每个 Task 绑定独立 WorkspaceRuntime。参考 `docs/CodePilot第二周开发策略说明.md` §五
- ⚠️ **方案 B**: 如果短期不需要多 Workspace，暂时保留全局 Workspace，但标记为技术债务
- ❌ **不建议**: 继续在现有全局 Workspace 上添加功能

**影响范围**: `ToolSystem.cpp`, `FileTool.cpp`, `ShellTool.cpp`, `GitTool.cpp`, `AgentLoop.cpp` (ToolContext 构造处)

### 4.2 Global / Session 概念 — 实验代码，建议重定义

**当前状态**:

```
sessions 表    → 只有 id, title, created_at (无消息内容)
globals 表     → KV 存储
global_context → 过渡表，在 global 上加任务级上下文快照
```

**问题**:
- `Session` 只是任务分组标签，不是真正的聊天会话（无 messages 表）
- `Global` 想做 KV 共享状态，但和 Task 没有联动（无"全局上下文检索"）
- `global_context` 表是临时妥协 — v2.3 将 Task 归属从 `session_id` 改为 `global_id`，但语义仍模糊
- **没有 Project 概念** — 缺少最上层的组织单位

**建议**:
- ✅ **方案 A (推荐)**: 引入 Project 概念，建立清晰层次：

```
Project (项目)
├── Workspace (文件工作区)
├── Sessions (对话历史)
│   ├── Messages (消息)
│   ├── Tasks (任务)
│   ├── Plan (计划)
│   └── Summary (摘要)
└── Long-Term Memory (长期记忆)
```

参考 `docs/CodePilot 本地工作区、长对话与文件上下文改造计划.md` §三

- ⚠️ **方案 B**: 保持现状，先将 Global 定位为"项目级 KV 补充"，Session 仅做任务分组，待下一阶段重构
- ❌ **不建议**: 继续新增 `global_context` 类型的过渡表

## 九、"公司"比喻设计意图说明

整个系统的设计隐喻是**一个公司**，目的是让配置 100% 驱动行为，无需重编译修改 loop 逻辑：

```
甲方 (用户) ──输入──→ 公司 ──输出──→ 用户

公司内部:
├── 员工花名册 (config/experts.json) — 自动登记所有员工及特长
├── 项目经理 (Planner, is_entry=true) — 拿到任务，制定计划
├── 执行员工 (Executor) — 按计划执行，操作公司帐本
├── 质检员 (Reviewer) — 审核产出，不合格打回
├── 汇报员 (Summarizer) — 向甲方交付最终成果
├── 公司帐本 (Global Context) — 跨任务共享状态
├── 个人草稿 (sessionHistory) — 员工内心独白
├── 项目规划 (PlanManager) — 动态数据结构，全员可维护
├── 聊天群 (Summary) — 关键上下文传递
└── 验收通道 (double_check) — 可选的多层审核

公司架构 100% 由甲方 (用户) 通过配置文件决定:
- 有没有规划员? 有没有质检员? 几层验收?
- 每个员工能用什么工具?
- 汇报风格是什么?
```
