# AI 编程智能体技术栈说明文档

## 1. 文档说明

本文档用于说明 AI 编程智能体项目的技术栈选型，包括后端服务、TUI 前端、数据存储、通信协议、工具调用、权限控制、日志记录、部署方式等方面的技术方案。

本项目目标是开发一个面向本地开发场景的 AI 编程智能体，能够理解用户输入的编程任务，支持多轮对话、上下文管理、工具调用、文件读写、Shell 命令执行、Web 搜索、Skill 插件、MCP 服务接入、执行日志记录、危险操作权限确认以及 TUI 界面实时展示。

系统整体采用前后端分离架构：

```text
TUI Client
   ↓ REST API / SSE
Agent Server
   ↓
Agent Core / Tool System / SQLite / Workspace / LLM / MCP
```

## 3. 技术选型原则

本项目的技术栈选择遵循以下原则：

### 3.1 与项目功能高度匹配

项目需要实现本地文件读写、Shell 命令执行、子进程管理、Git 操作、TUI 终端界面等能力。C++ 对系统资源、进程、文件和性能控制较强，适合作为项目主语言。

### 3.2 保持技术栈统一

后端服务、Agent 核心模块、工具系统和 TUI 客户端均可使用 C++ 实现，减少多语言协作成本，降低项目复杂度。

### 3.3 支持前后端分离

系统采用 TUI Client + Agent Server 的前后端分离模式。普通请求通过 REST API 完成，任务执行过程通过 SSE 实时推送给前端。

### 3.4 支持本地部署和演示

项目使用 Docker 部署后端服务，便于统一运行环境、简化启动命令、隔离 workspace 目录，并保证 SQLite 数据持久化。

### 3.5 方便后续扩展

系统后续可能扩展 Skill 插件、MCP 服务、更多工具类型、权限策略、历史回放能力。因此技术栈应支持模块化设计和插件式扩展。

## 4. 后端技术栈

### 4.1 开发语言：C++20

后端采用 C++20 作为主要开发语言。

选择原因：

1. 适合系统级开发，方便处理文件系统、进程管理、Shell 命令执行等能力。
2. 性能较好，适合处理实时事件流、日志记录和任务调度。
3. 与 TUI 客户端技术栈统一，减少跨语言维护成本。
4. 支持现代 C++ 特性，例如智能指针、线程、异步、结构化绑定、`std::filesystem` 等。

后端主要实现内容包括：

```text
会话管理
消息管理
任务调度
Agent 执行循环
上下文管理
工具调用
权限控制
日志记录
SSE 事件推送
SQLite 数据持久化
LLM API 调用
MCP 服务调用
```

---

### 4.2 后端 Web 框架：Drogon

后端 Web 服务建议使用 Drogon。

主要作用：

```text
实现 HTTP REST API
实现 SSE 事件流接口
处理 JSON 请求和响应
管理路由和 Controller
支持异步请求处理
```

项目中需要实现的接口包括：

```text
/api/v1/sessions
/api/v1/workspaces
/api/v1/tasks
/api/v1/tasks/{task_id}/events
/api/v1/permissions
/api/v1/tools
/api/v1/skills
/api/v1/mcp
/api/v1/config
```

Drogon 适合该项目的原因：

1. C++ 原生 Web 框架，和项目主语言一致。
2. 性能较好，适合实现 Agent Server。
3. 支持异步处理，适合任务执行和事件推送。
4. 可以实现 REST API 和 SSE 长连接。
5. 项目结构清晰，适合 Controller + Service + Repository 分层。

### 4.3 JSON 处理：nlohmann/json

系统前后端通信、工具参数、事件 metadata、配置文件都采用 JSON 格式，因此需要统一的 JSON 处理库。

推荐使用 nlohmann/json。

主要用途：

```text
解析 HTTP 请求体
生成 HTTP 响应体
解析 agent.json 配置文件
保存工具调用参数
保存工具调用结果
保存 SSE 事件 metadata
保存任务 plan
保存 MCP 调用参数
```

示例场景：

```json
{
  "session_id": "session_001",
  "workspace_id": "workspace_001",
  "input": "帮我修复这个 C++ 项目的编译错误",
  "options": {
    "auto_run_safe_commands": true,
    "require_permission_for_file_write": true,
    "max_steps": 20
  }
}
```

### 4.4 HTTP Client：libcurl / cpr

Agent Server 需要调用外部服务，例如：

```text
大模型 API

远程文档接口
```

推荐优先使用 libcurl。如果希望代码更简洁，也可以在 libcurl 之上使用 cpr 作为封装。

主要用途：

```text

调用 Embedding API
调用 Web Search API
调用远程 MCP 服务
处理 HTTP 超时
处理请求头和认证信息
```

项目需要记录多种日志：

```text
系统启动日志
HTTP 请求日志
Agent 执行日志
工具调用日志
权限确认日志
Shell 输出日志
```

日志建议分为两类：

### 5.2 TUI 与后端通信

TUI Client 与 Agent Server 之间采用两种通信方式：

#### 1. REST API

用于普通操作，例如：

```text
创建会话
创建工作区
创建任务
查询任务详情
查询权限请求
同意或拒绝权限
查询文件变更
查询历史回放
```

#### 2. SSE 事件流

用于实时展示 Agent 执行过程，例如：

```text
任务创建
任务规划
Agent 回复
工具开始执行
工具输出
工具执行完成
请求权限确认
文件发生变更
任务完成
任务失败
```

SSE 事件示例：

```text
event: tool_output
data: {"type":"tool_output","content":"undefined reference to Config::load","metadata":{"stream":"stderr"}}
```

## 6. 数据存储技术栈

### 6.1 数据库：SQLite

本项目采用 SQLite 作为本地数据库。

SQLite 适合该项目的原因：

1. 项目以本地开发场景为主，不需要复杂数据库服务。
2. 部署简单，无需单独安装数据库服务器。
3. 适合存储会话、任务、事件、日志等结构化数据。
4. 可以通过 Docker volume 持久化到宿主机。
5. 方便项目演示和本地运行。

主要数据表包括：

```text
sessions
messages
workspaces
tasks
task_events
tool_calls
permission_requests
file_changes
execution_logs
```

---

### 6.2 数据访问层

建议封装 Repository 层，不要在业务逻辑中直接拼接 SQL。

推荐 Repository 设计：

```text
SessionRepository
MessageRepository
WorkspaceRepository
TaskRepository
EventRepository
ToolCallRepository
PermissionRepository
FileChangeRepository
ExecutionLogRepository
```

这样可以降低业务逻辑与数据库实现的耦合，后续如果需要从 SQLite 切换到 PostgreSQL，也更容易迁移。

## 7. Agent Core 技术设计

Agent Core 是系统核心模块，负责真正执行用户任务。

推荐模块划分：

```text
agent/
├── AgentRunner
├── Planner
├── ContextManager
├── LLMClient
├── ToolRouter
├── ToolRegistry
├── PermissionManager
├── EventBus
└── TaskStateManager
```

各模块职责如下：

| 模块                | 职责                      |
| ----------------- | ----------------------- |
| AgentRunner       | 控制 Agent 执行循环           |
| Planner           | 根据用户任务生成执行计划            |
| ContextManager    | 管理多轮对话上下文、任务上下文、工具结果上下文 |
| LLMClient         | 调用大模型 API               |
| ToolRouter        | 根据 Agent 决策选择合适工具       |
| ToolRegistry      | 注册和管理工具                 |
| PermissionManager | 判断工具调用是否需要用户确认          |
| EventBus          | 写入事件并推送 SSE             |
| TaskStateManager  | 管理任务状态流转                |

Agent 执行流程：

```text
用户创建任务
↓
AgentRunner 加载 Session 和 Workspace
↓
ContextManager 构造上下文
↓
Planner 生成执行计划
↓
LLMClient 调用大模型
↓
ToolRouter 选择工具
↓
PermissionManager 检查风险
↓
工具执行
↓
EventBus 记录并推送事件
↓
Agent 根据结果继续执行
↓
任务完成或失败
```

## 8. 工具系统技术栈

### 8.1 工具注册机制

系统应设计统一的 Tool 接口，所有工具都通过 ToolRegistry 注册。

推荐工具接口：

```cpp
class Tool {
public:
    virtual std::string name() const = 0;
    virtual std::string description() const = 0;
    virtual RiskLevel risk_level(const json& arguments) const = 0;
    virtual ToolResult execute(const ToolContext& context, const json& arguments) = 0;
    virtual ~Tool() = default;
};
```

```text
process/

├── CommandPolicy
├── OutputStreamer
```

Shell 执行模块需要支持：

```text

实时推送输出
支持 timeout
支持取消任务时终止子进程
识别危险命令
```

危险命令示例：

```text
mkfs

curl | sh

```

## 10. 权限与安全技术栈

权限模块用于控制高风险操作，防止 Agent 误删文件、越权访问系统目录或执行危险命令。

推荐模块：

```text
security/
├── PathGuard
├── CommandRiskAnalyzer
├── PermissionService
├── PolicyStore
└── AuditLogger
```

### 10.1 路径安全

路径安全规则：

```text
Agent 默认只能访问 workspace 内部文件
禁止通过 ../ 访问 workspace 外部路径
禁止通过软链接逃逸 workspace
禁止读取系统敏感路径
限制最大文件大小
禁止读取二进制文件
```

### 10.2 命令安全

命令安全规则：

```text
安全命令可以自动执行
中风险命令需要根据配置判断是否确认
危险命令必须用户确认
阻断级命令直接拒绝执行
```

### 10.3 文件写入安全

文件写入规则：

```text
写文件需要用户确认
删除文件需要用户确认
应用 patch 需要用户确认
批量修改需要展示 diff 预览
确认后才能真正写入
```

## 11. MCP 技术栈

MCP 用于接入外部工具服务，使 Agent 可以调用更多能力。

第一版建议支持：

```text
MCP over stdio
JSON-RPC 2.0
工具发现
工具调用
调用结果解析
权限拦截
调用日志记录
```

MCP 模块建议划分：

```text
mcp/
├── McpClient
├── McpServerManager
├── McpToolAdapter
├── JsonRpcMessage
└── McpPermissionBridge
```

MCP 调用流程：

```text
Agent 请求调用 MCP 工具
↓
ToolRouter 路由到 mcp.call
↓
PermissionManager 判断风险
↓
McpClient 发送 JSON-RPC 请求
↓
MCP Server 返回结果
↓
结果写入 ToolCall 和 Event
↓
Agent 继续执行
```

## 12. Skill 插件技术栈

Skill 是系统内部可扩展能力，用于封装常见开发任务。

第一版可以内置：

```text
cmake_build
cpp_test
git_diff_review
dependency_check
compile_error_fix
```

Skill 可以使用 JSON 描述：

```json
{
  "name": "cmake_build",
  "description": "配置并构建 CMake 项目",
  "tools": ["shell.run"],
  "arguments_schema": {
    "build_dir": "string",
    "build_type": "string"
  }
}
```

Skill 的作用是把多个底层工具调用组合成一个更高层的能力。例如 `cmake_build` 可以内部调用：

```text
file.list
shell.run: cmake -S . -B build
shell.run: cmake --build build
```

本项目最终推荐技术栈如下：

```text

Drogon + nlohmann/json + SQLite + spdlog + libcurl

FTXUI + HTTP Client + SSE Client + nlohmann/json

SQLite

REST API + SSE + JSON

JSON-RPC 2.0 over stdio

CMake

依赖管理：
vcpkg

GoogleTest + CTest

Docker + docker-compose
```
