## 2. 核心目标达成评估：完全为主循环服务 ✅

**Agent::executeTask() 现在仅依赖 4 个 `getInstance()` 单例**：

```
Agent::executeTask()
  ├── ToolSystem::getInstance()           → callToolWithPermission()
  ├── DataAccessFacade::getInstance()     → appendLog / saveEvent / saveToolCall / recordFileChange
  ├── LlmClientFacade::getInstance()      → chat / isAvailable
  └── SSEGateway::getInstance()           → pushDialog / pushStatus / pushDebug
```

**不再出现**：
- ❌ `sqlite3*` 指针
- ❌ `LogRepository(db_).createLog(...)` 等直接 Repository 构造
- ❌ `llmClient_->chat(...)` （成员变量依赖）
- ❌ `ToolSystem::getInstance().eventBus().publish(...)` （EventBus 直调）
- ❌ `setDb()` / `setLlmClient()` 等外部注入方法

**验证方法**：`Agent.cpp` 中搜索 `Repository(` — 零匹配。搜索 `db_` — 零匹配。搜索 `eventBus` — 零匹配。搜索 `llmClient_` — 零匹配。

## 3. 剩余外部依赖与潜在问题清单

### A 类：必须协作才能生效的（阻塞性）

| 依赖方 | 事项 | 影响 |
|--------|------|------|
| **刘子恒（前端）** | SSE 三通道分流：按 `metadata.channel` 将 `"debug"` 内容移出对话框 | 当前 debug 信息涌入对话面板，用户体验差但功能不受阻 |
| **曾嘉睿** | `AgentRunnerFacade` 实现（优先级 5），封装 Core Loop 为单入口 `run()` | Agent Core 已就绪可直接写，无需等任何门面 |
| **郑嘉娴** | `TaskOrchestrator` 实现（优先级 6），统一管理异步线程+重试+取消 | 当前 TaskController 的 `std::thread::detach()` + `worker_db` 是临时方案 |

### B 类：当前代码中的遗留问题（非阻塞性）

| 位置 | 问题 | 建议 |
|------|------|------|
| `AppBootstrap::initialize_database()` | 内联了 150 行 SQL schema（sessions/tasks/workspaces 等建表语句），与 `DataAccessFacade::createAllTables()` 重复 | 应统一到 DataAccessFacade |
| `AgentService::getAgent()` | 返回 `Agent*` 裸指针，可能被外部用于直接调 `setDb()`/`setLlmClient()`（现已标记 `[[deprecated]]`） | 保留兼容，待曾嘉睿重写后移除 |
| `TaskController::createTask()` | 仍手动管理 `sqlite3* db` 连接，通过 `TaskService(db)` 操作 | 建议改为 `DataAccessFacade::createTask()` + `updateTaskStatus()` |
| `TaskController::listToolCalls/::listEventHistory` | 直接构造 `ToolCallRepository(db)` / `EventRepository(db)` 查询 | 应改为 `DataAccessFacade::getToolCallsByTaskId()` / `getEventsByTaskId()` |
| `AgentService::runTask()` | 每次调用创建新的 `RoleRegistry`+`Planner`+`Agent` 实例 | 性能可接受，但可考虑 pool/reuse |
| `LlmClientFacade::init()` | 调用时机在 `AppBootstrap` 中排在 `DataAccessFacade` 和 `SSEGateway` 之后 | 顺序正确，无循环依赖 |

### C 类：不需要任何外部配合的（自完备项）

以下功能已经自完备，不需要任何其他人修改代码：

| 项 | 说明 |
|----|------|
| 8 表持久化 | `DataAccessFacade` 单例覆盖全部数据表 |
| 多 Provider LLM | `LlmClientFacade` + `llm.json` + `llm.local.json` |
| SSE 兼容推送 | `SSEGateway` 保持 `event:` 标签不变，仅 `metadata.channel` 新增 |
| 配置清理 | `agent.json` 死配置已移除 |

### 总结

**合并风险**：低。所有变更都是增量添加（新增 facade 目录 + 8 个新文件），已有文件只做减法（移除依赖而非新增接口），合并冲突概率极低。

**需要配合的动作**（按优先级）：
1. 刘子恒前端按 `metadata.channel` 分流（非紧急，当前可正常运行）
2. 郑嘉娴/曾嘉睿后续封装时统一调用 `DataAccessFacade` 方法替代保留的 `TaskService` 直调
3. AppBootstrap 中的 schema 重复问题统一到 `DataAccessFacade`
