## 当前待处理

### 🟡 P1 — 影响可靠性

#### P1-01: SSE 双通道架构未收敛

- **位置**: `HttpServer.cpp:584-699` vs `SSEGateway.cpp:319-476`
- **描述**: HttpServer 的 `streamTaskEvents()` 直接订阅 EventBus 处理 SSE 长连接，绕过了 SSEGateway 的客户端注册机制。导致 `SSEGateway::liveClients_` 列表永远为空，`getConnectionCount()` 永远返回 0。
- **影响**: 功能正常运行（EventBus 仍能推送所有事件到前端），但 SSEGateway 的统一中介设计被架空
- **修复方向**: 将 HttpServer 的 SSE 处理迁移到 SSEGateway，统一使用 `streamTaskEvents(SendCallback, taskId)` 接口
- **状态**: 待处理

#### P1-02: SSEGateway client removal 比较逻辑不可靠

- **位置**: `SSEGateway.cpp:469-473`
- **描述**: `streamTaskEventsImpl()` 中使用 `std::function::target_type()` 来比对 callback 以注销客户端。不同类型的 lambda 可能有相同的 target_type，导致误删或不删。
- **影响**: 当前为死代码（因为 P1-01 导致 liveClients_ 无客户端），无实际影响。但修复 P1-01 后此问题将变为真实 bug。
- **修复方向**: 改用唯一 ID（如递增序号）标识客户端，而非依赖 `target_type()` 比较
- **状态**: 待处理（P1-01 的依赖项）

#### P1-03: AgentOrchestrator 任务完成后不清理 map

- **位置**: `AgentOrchestrator.cpp` — `activeTasks_`, `cancelFlags_`, `taskThreads_` 三个 std::unordered_map
- **描述**: `runTaskThread()` 和 `runContinueThread()` 中的线程 detach 后，对应的 map entry 从不 erase。每个已完成任务约占用 1-2KB 内存。
- **影响**: 长时间运行数千个任务后可能导致内存增长。正常使用场景下影响极小。
- **修复方向**: 在 `runTaskThread()` 末尾调用一个清理方法，安全移除已完成任务的 map entry。需要注意 detach 的线程访问同步。
- **状态**: 待处理

#### P1-04: AgentLoop 构造函数重复调用 AgentConfiguration::init()

- **位置**: `AgentLoop.cpp:19` — `AgentConfiguration::getInstance().init(configPath)`
- **描述**: `AgentOrchestrator::init()` 已调用过 `AgentConfiguration::init()`，每个 AgentLoop 创建时又调用一次。虽然是幂等的，但属于冗余操作。
- **影响**: 零 — init() 是幂等的，第二次调用本质上无操作
- **修复方向**: 移除 AgentLoop 构造函数中的 init() 调用，或改为 `reconfigure()`
- **状态**: 待处理

### 🟢 P2 — 优化建议

#### P2-01: continueTask 不解析 options 字段

- **位置**: `TaskController.cpp:213-247` — `continueTask()` 方法
- **描述**: 创建任务的 POST body 文档中定义了 `options` 字段（execution_mode, max_steps 等），但 `continueTask()` 没有解析和使用这些选项
- **影响**: 前端传入的 options 被静默忽略
- **修复方向**: 在 continueTask 中添加 options 解析逻辑，传递给 AgentOrchestrator
- **状态**: 待处理

#### P2-02: taskId 拼写不一致

- **位置**: 多处（`TaskContext.h` 使用 `taskId`, Plan/MessageBus 等处使用 `task_id`）
- **描述**: 代码库中混用 camelCase 和 snake_case 的 `taskId` / `task_id` 命名
- **影响**: 代码风格不一致，不影响功能
- **修复方向**: 统一为一种命名风格（建议 C++ 层用 camelCase，JSON/API 层用 snake_case）
- **状态**: 待处理

#### P2-03: PlanManager.history() 线程安全不完整

- **位置**: `PlanManager.h:46` — `history()` 方法
- **描述**: `history()` 返回 `std::vector<PlanSnapshot>` 的拷贝，但拷贝过程中没有加锁，可能在并发场景下读到不一致的快照
- **影响**: 当前 Expert Chain 是串行的，无实际并发风险。但如果未来支持并行 Expert，需要修复。
- **修复方向**: 对 `history_` 的读取加锁
- **状态**: 待处理

#### P2-04: constructPrompt 中的 contextTemplate 占位符无文档化枚举

- **位置**: `PromptBuilder.cpp` — `constructPrompt()` 方法
- **描述**: contextTemplate 支持的占位符（`{role}` `{goal}` `{plan}` `{summary}` `{tools_desc}` `{tag_protocol}` `{output_hint}` `{rounds_left}` `{session}`）没有在一个集中的头文件或文档中枚举定义
- **影响**: 配置文件编写者不知道有哪些可用占位符，只能看源码或示例
- **修复方向**: 在 `PromptBuilder.h` 中增加占位符常量定义，或在 `docs/Expert配置说明文档.md` 中补充完整列表
- **状态**: 待处理

#### P2-05: RouteEngine.resolveForFail 存在但未被 AgentLoop 调用

- **位置**: `RouteEngine.h:25` — `resolveForFail()` 声明
- **描述**: `resolveForFail()` 是一个静态方法，用于仅根据 `onFailRoute` 解析兜底路由。但 `AgentLoop::runExpertChain()` 在检测到 `<fail>` 时使用的是完整的 `resolve()` 方法
- **影响**: 死代码，不影响功能
- **修复方向**: 确认是保留备用还是移除，如果保留加注释说明用途
- **状态**: 待处理

#### P2-06: ~~tools.json 注册了未实现的 mcp.call 工具~~ ✅ 已关闭

- **位置**: `config/tools.json:172-182` — `mcp.call` 工具定义
- **描述**: `mcp.call` 在 tools.json 中定义但无对应实现，LLM 可能诱导调用不存在的工具
- **修复**: 2026-07-13 从 tools.json 移除 mcp.call 条目，待 MCP 实现后再加回
- **状态**: ✅ 已关闭
- **发现日期**: 2026-07-13
- **修复日期**: 2026-07-13

#### P2-07: ~~权限拒绝后 LLM 收到的失败语义不够精确~~ ✅ 已关闭

- **位置**: `application/ToolSystem.cpp` — `callToolWithPermission()` / `AgentLoop.cpp` — `<cmd>` 处理
- **描述**: 权限拒绝/工具未找到/not_in_list 等场景缺乏精确语义标签，LLM 无法区分失败类型
- **修复**: 2026-07-13 引入五级精确标签体系：
  - `[NOT_AVAILABLE]` — Expert 工具白名单拦截，附带可用工具列表
  - `[TOOL_NOT_FOUND]` — 工具未注册
  - `[BLOCKED]` — 系统永久阻止（含建议使用替代方案）
  - `[PERMISSION_DENIED]` — 用户拒绝（含建议用无权限替代方案或等批准）
  - `[PERMISSION_EXPIRED]` — 等待超时（120秒未响应）
  - 全部附带 `failure_reason` 元数据字段
- **状态**: ✅ 已关闭
- **发现日期**: 2026-07-13
- **修复日期**: 2026-07-13

#### P2-08: PermissionController 构造函数 `databasePath_` 参数未使用

- **位置**: `api/controllers/PermissionController.h:10` / `PermissionController.cpp:96`
- **描述**: 构造函数接受 `databasePath` 参数但从未被使用，所有操作通过 `ToolSystem::permissionManager()` 走内存。HttpServer 中传入 `config_.databasePath` 被静默忽略。
- **影响**: 代码异味，不影响功能。⚠️ **注意**：PermissionController 本身不是死代码——它是旧前端批准/拒绝权限的唯一 HTTP API 入口，通过 `PermissionManager::resolvePermission()` 唤醒阻塞的 Agent 线程，不可删除。
- **修复方向**: 移除 `databasePath_` 成员和构造函数参数，简化为无参构造函数
- **状态**: 待处理
- **发现日期**: 2026-07-13

#### P2-09: Debugger API 端点缺失

- **位置**: `api/HttpServer.cpp` — `handleRequest()` 方法
- **描述**: ToolSystem 已初始化 Debugger 子系统，`Debugger.h` 定义了完整的断点控制 API，但 HttpServer 中没有任何 `/api/v1/debug/*` 路由注册。调试器功能完全不可用。
- **影响**: 调试器功能不可用——用户无法设置断点、单步执行、修改参数/结果
- **修复方向**: 创建 `DebugController` 并注册 9 个 API 端点到 HttpServer
- **状态**: ✅ 已关闭
- **关闭原因**: 2026-07-13 完成修复：创建 `DebugController.h/.cpp`，注册 9 个端点，更新 CMakeLists.txt
- **发现日期**: 2026-07-13
- **修复日期**: 2026-07-13
