# CodePilot 前后端 REST API 数据传输协议规范

> **版本**: v1.0
> **更新**: 2026-07-14
> **适用范围**: `HttpServer` ROUTE 宏 → Controller 链路的全部 API 端点
> **核心原则**: 定义一份唯一权威的接口契约，消除当前反复出现的路径解析、编码、路由匹配问题。

---

## 一、架构背景与问题根源

### 1.1 当前架构

```
浏览器 / CLI (ApiClient)
    │  HTTP (cpp-httplib)
    ▼
HttpServer::run()  ───  httplib::Server 路由匹配
    │
    │  ROUTE 宏组装「伪 HTTP 请求字符串」
    ▼
Controller::ctrlMethod(request)
    │  解析 request 字符串提取参数
    ▼
    │  返回完整 HTTP 响应字符串
    │  (HTTP/1.1 200 OK\r\n...\r\n\r\n{json})
    ▼
HttpServer::respondJson()
    │  extract_json_body() 剥离 HTTP 头
    │  → 纯 JSON body 返回客户端
    ▼
客户端
```

### 1.2 已发现的所有共性 bug 及根因

| # | Bug 现象 | 根因 | 类别 |
|---|---------|------|------|
| 1 | `/workspace files id` → "workspace_id is required" | `WorkspaceFileController::request_target()` 期望 HTTP 版本号但伪请求没有 | **请求格式** |
| 2 | `/workspace files id` → "invalid UTF-8 byte 0xDE" | `std::filesystem::path` 返回 GBK 编码的文件名，直接拼入 JSON | **编码** |
| 3 | `/perm list` → "permission request not found" | `ApiClient` 请求 `/permissions/pending`，服务端无此路由 | **路由匹配** |
| 4 | `/perm approve` → "path must end with /approve" | 正则 `/permissions/(id)` 在精确路由 `/permissions/approve-first` 之前匹配 | **路由顺序** |
| 5 | `POST body` 解析为空白 | ROUTE 宏缺少 `\r\n\r\n` 分隔符 | **请求格式** |

**根本原因**：缺少一份定义以下内容的权威协议文档：
1. 伪请求字符串的**确切格式**
2. Controller 解析函数的**输入/输出契约**
3. 字符编码的**强制规则**
4. 路由注册的**优先级规则**

## 二、伪请求字符串格式规范（ROUTE 宏 → Controller）

### 2.1 格式定义

ROUTE 宏向 Controller 传入的 `request` 参数是一个字符串，**不包含 HTTP 版本号**，格式为：

```
{METHOD} {PATH}[?{QUERY}]\r\n\r\n[{BODY}]
```

**关键规则**：
- ❌ **不包含** `HTTP/1.1` 版本标记
- ✅ 路径和可选的 query string 后紧跟 `\r\n\r\n`
- ✅ POST/PUT/PATCH 时，`\r\n\r\n` 后紧跟 JSON body
- ✅ GET/DELETE 时，`\r\n\r\n` 后为空（无 body）

### 2.2 标准示例

| HTTP 方法 | 伪请求字符串 |
|-----------|------------|
| `GET /api/v1/sessions` | `GET /api/v1/sessions\r\n\r\n` |
| `GET /api/v1/sessions?page=1` | `GET /api/v1/sessions?page=1\r\n\r\n` |
| `GET /api/v1/sessions/sess_abc` | `GET /api/v1/sessions/sess_abc\r\n\r\n` |
| `POST /api/v1/sessions` | `POST /api/v1/sessions\r\n\r\n{"title":"hello"}` |
| `PUT /api/v1/sessions/sess_abc` | `PUT /api/v1/sessions/sess_abc\r\n\r\n{"alias":"my"}` |
| `DELETE /api/v1/sessions/sess_abc` | `DELETE /api/v1/sessions/sess_abc\r\n\r\n` |

### 2.3 ROUTE 宏 argExpr 写法

```cpp
// 无参数（仅路径）
ROUTE0(Get, "/api/v1/tasks/active", TaskController, listActiveTasks);

// 带 query string → 必须传入 build_query_string(req.params)
ROUTE(Get, "/api/v1/sessions", SessionController, listSessions,
      "GET /api/v1/sessions?" + build_query_string(req.params));

// 带路径参数 → 用 req.matches[1].str()
ROUTE(Get, R"(/api/v1/sessions/([a-zA-Z0-9\-_]+))", SessionController,
      getSession, "GET /api/v1/sessions/" + req.matches[1].str());

// 带 body → 拼接 req.body
ROUTE(Post, "/api/v1/sessions", SessionController, createSession,
      "POST /api/v1/sessions \r\n\r\n" + req.body);

// 带路径参数 + body
ROUTE(Put, R"(/api/v1/sessions/([a-zA-Z0-9\-_]+))", SessionController,
      updateSession,
      "PUT /api/v1/sessions/" + req.matches[1].str() + " \r\n\r\n" + req.body);
```

### 2.4 ✅ 正确的 Controller 解析实现参考

```cpp
// 提取路径中 /api/v1/xxx/{id} 的 id 段
std::string extract_path_segment(const std::string &request,
                                 const std::string &prefix) {
  const std::size_t request_line_end = request.find("\r\n");
  const std::string request_line = request.substr(0, request_line_end);
  const std::size_t method_end = request_line.find(' ');
  if (method_end == std::string::npos) return "";
  const std::size_t path_start = method_end + 1;
  const std::size_t prefix_pos = request_line.find(prefix, path_start);
  if (prefix_pos == std::string::npos) return "";
  const std::size_t segment_start = prefix_pos + prefix.size();
  const std::size_t segment_end =
      request_line.find_first_of("? ", segment_start);
  if (segment_end == std::string::npos) {
    return request_line.substr(segment_start);
  }
  return request_line.substr(segment_start, segment_end - segment_start);
}

// 提取 POST/PUT body
std::string request_body(const std::string &request) {
  const std::size_t body_pos = request.find("\r\n\r\n");
  if (body_pos == std::string::npos) return "";
  return request.substr(body_pos + 4);
}
```

> **⚠️ 强制规则**：所有新增 Controller **必须**使用上述标准解析函数。
> `WorkspaceFileController` 的自建解析逻辑需要改造为统一解析。

## 三、JSON 响应格式规范

### 3.1 统一响应结构

所有 API 返回的 JSON body 必须包含顶层 `success` 字段：

```json
// 成功响应
{
  "success": true,
  "data": { ... }
}

// 失败响应
{
  "success": false,
  "error": {
    "code": "ERROR_CODE",
    "message": "人类可读的错误描述"
  }
}
```

### 3.2 标准错误码

| 错误码 | HTTP 状态 | 含义 |
|--------|----------|------|
| `INVALID_REQUEST` | 400 | 请求参数缺失或无效 |
| `NOT_FOUND` | 404 | 资源不存在 |
| `DATABASE_ERROR` | 500 | 数据库操作失败 |
| `PERMISSION_NOT_FOUND` | 404 | 权限请求不存在 |
| `NO_PENDING` | 404 | 无待处理权限 |
| `ALREADY_RESOLVED` | 400 | 权限已处理 |
| `INTERNAL_ERROR` | 500 | 内部错误 |

### 3.3 Controller 返回值格式（⚠️ 可能包含 HTTP 头）

旧式 Controller 返回**完整 HTTP 响应字符串**（含 HTTP 头），`HttpServer::respondJson()` 内部调用 `extract_json_body()` 剥离头部。

```
HTTP/1.1 200 OK\r\n
Content-Type: application/json; charset=utf-8\r\n
Content-Length: 42\r\n
\r\n
{"success":true,"data":{...}}
```

**新 Controller 应直接返回纯 JSON body**，使用 `respondJson` 辅助函数。

## 四、字符编码强制规范

### 4.1 硬性规则

| 场景 | 规则 | 违规处理 |
|------|------|----------|
| JSON 字符串 | **必须是合法 UTF-8** | 非法字节序列 → 替换为 `\uFFFD` |
| `std::filesystem::path` 转字符串 | **必须先转 UTF-8** | 使用 `path.u8string()` |
| 文件内容读取 | 检测编码，非 UTF-8 标记 | 标记 `encoding` 字段 |
| 二进制文件内容 | **禁止直接放入 JSON** | Base64 编码或返回 415 |
| HTTP Content-Type | 必须声明 `charset=utf-8` | — |

### 4.2 标准 UTF-8 转换函数

```cpp
#include <filesystem>
#include <string>

// Windows: std::filesystem::path → UTF-8
std::string path_to_utf8(const std::filesystem::path &p) {
#ifdef _WIN32
  return p.u8string();  // C++20: 返回 UTF-8 std::string
#else
  return p.string();
#endif
}
```

### 4.3 当前 bug 修复方案

`WorkspaceFileController::getTree()` 中：

```cpp
// ❌ 错误：直接使用 entry.name（可能是 GBK）
items.push_back({{"name", entry.name}, ...});

// ✅ 正确：转换为 UTF-8
items.push_back({{"name", path_to_utf8(entry.name)}, ...});
```

**涉及文件**：
- `backend/agent-server/api/controllers/WorkspaceFileController.cpp`
- `workspace.json` 中返回的 `path` 字段

## 五、HTTP 方法规范

| 方法 | 用途 | 体格式 | 幂等 |
|------|------|--------|------|
| GET | 查询 | 无 body，参数在 query string 或路径中 | 是 |
| POST | 创建/动作 | `Content-Type: application/json` | 否 |
| PUT | 全量更新 | `Content-Type: application/json` | 是 |
| DELETE | 删除 | 无 body，ID 在路径中 | 是 |

### CLI ApiClient 对应方法

```cpp
// ApiClient.h 中每个 HTTP 方法必须具备对应的 doXxx 方法：
std::string doGet(const std::string &path);
std::string doPost(const std::string &path, const std::string &body);
std::string doPut(const std::string &path, const std::string &body);
std::string doDelete(const std::string &path);
```

## 六、路由注册规范

### 6.1 优先级规则（⚠️ 关键：已引发多次 bug）

**精确路由必须写在正则路由前面！**

```cpp
// ✅ 正确顺序
ROUTE(Post, "/api/v1/permissions/approve-first", ...);          // 精确路由在前
ROUTE(Post, R"(/api/v1/permissions/([a-zA-Z0-9\-_]+))", ...);  // 正则路由在后

// ❌ 错误顺序（会导致 /approve-first 被正则捕获为 perm ID）
ROUTE(Post, R"(/api/v1/permissions/([a-zA-Z0-9\-_]+))", ...);  // 正则路由在前
ROUTE(Post, "/api/v1/permissions/approve-first", ...);          // 永远无法匹配
```

### 6.2 路径参数正则

路径参数使用正则捕获组 `([a-zA-Z0-9\-_]+)`：

| 模式 | 匹配示例 |
|------|---------|
| `([a-zA-Z0-9\-_]+)` | session_id, workspace_id, task_id |
| 固定路径 | `/api/v1/permissions/approve-first` |

## 七、SSE 事件流协议

### 7.1 端点

```
GET /api/v1/tasks/{task_id}/events
```

### 7.2 帧格式（完全标准 SSE）

```
event: {event_type}
data: {单行 JSON}

```

- `event:` 行 → 事件类型（如 `agent_message`、`tool_started`）
- `data:` 行 → 单行 JSON，包含六字段：`id, task_id, type, content, metadata, created_at`
- 空行 → 帧分隔符

### 7.3 心跳

每 3 秒发送 SSE 注释行 `: ping\n\n`，客户端应忽略。

## 九、协议合规检查清单

实现新 API 或修改现有 API 时，必须检查：

- [ ] ROUTE 宏 argExpr 格式符合 §2.2 标准模板
- [ ] 精确路由在正则路由之前（§6.1）
- [ ] JSON body 响应格式 `{"success": bool, "data": {...}}` （§3.1）
- [ ] `Content-Type: application/json; charset=utf-8` 已设置（§4.1）
- [ ] 文件路径/文件名已转换为 UTF-8（§4.2）
- [ ] Controller 使用标准解析函数（§2.4）
- [ ] CLI ApiClient 路径与服务端路由一致（§8）
- [ ] 编译通过，零错误
