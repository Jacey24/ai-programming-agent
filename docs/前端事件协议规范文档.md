## 一、REST API 协议基础（必读）

所有前端通过 REST API 与服务端通信。API 协议规范详见 **[前后端 REST API 数据传输协议规范](./前后端数据传输协议规范.md)**。以下是与事件消费直接相关的要点：

### 1.1 统一响应格式

```json
// 所有 REST API 返回
{ "success": true, "data": { ... } }
{ "success": false, "error": { "code": "...", "message": "..." } }
```

### 1.2 字符编码

- **所有 JSON 必须 UTF-8** — HTTP `Content-Type: application/json; charset=utf-8`
- **文件路径/文件名**：Windows 下 `std::filesystem::path` 必须用 `.u8string()` 转换
- **路由优先级**：精确路由必须写在正则路由之前

### 1.3 请求/响应管线

```
客户端 (ApiClient/EventSource)
    │  HTTP GET/POST/PUT/DELETE
    ▼
HttpServer → ROUTE 宏拼装「伪 HTTP 请求字符串」→ Controller
    │                           (无 HTTP 版本号！)
    │  Controller 返回 HTTP 响应字符串
    ▼
extract_json_body() 剥离 HTTP 头 → 纯 JSON 返回客户端
```

> ⚠️ 所有前端开发者必须注意：Controller 接收的请求字符串 **不含 HTTP 版本号**（如 `GET /api/v1/tasks/xxx\r\n\r\n`）。解析时不能依赖 `HTTP/1.1` 标记。

```
   └────────────┘  └──────────────┘  └──────────────┘
```

### 通路说明

| 通路 | 传输方式 | 适用前端 | 延迟 |
|------|---------|---------|------|
| **通路 A: EventBus 回调** | `std::function<void(EventData&)>` 内存回调 | C++ TUI / 同进程 GUI | <1ms |
| **通路 B: HTTP SSE** | `text/event-stream` over HTTP/1.1 | Web 前端 / 远程客户端 | ~10ms |
| **通路 C: REST 历史查询** | `GET /api/v1/tasks/:id/events/history` JSON | 所有前端（回放/刷新） | ~50ms |

**三种通路消费的是完全相同的 JSON 数据结构**——唯一的区别是传输层。

## 二、SSE 数据协议（核心规范）

### 2.1 SSE 帧格式

```
event: {event_type}
data: {json_payload}

```

- `event:` 行 = EventType 枚举的小写下划线形式（如 `agent_message`、`tool_started`）
- `data:` 行 = 单行 JSON（不含换行），六字段固定结构

### 2.2 JSON 数据体（六字段）

```json
{
  "id": "event_c0a80101_0001",
  "task_id": "t_abc123def",
  "type": "agent_message",
  "content": "已完成文件创建",
  "metadata": {
    "channel": "dialog",
    "expert": "executor",
    "streaming": false
  },
  "created_at": "2026-07-12T09:15:00Z"
}
```

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `id` | string | 是 | 全局唯一事件 ID，格式 `event_{ts}_{counter}` |
| `task_id` | string | 是 | 所属任务 ID |
| `type` | string | 是 | 事件类型（见 §2.3） |
| `content` | string | 是 | 事件内容（人类可读文本或结构化数据） |
| `metadata` | object | 是 | 扩展元数据（见 §2.4） |
| `created_at` | string | 是 | ISO 8601 UTC 时间戳 |

### 2.3 事件类型枚举（13 种）

| 事件类型 | 频道 | 含义 | 触发时机 |
|----------|------|------|----------|
| `task_created` | status | 任务已创建 | POST /api/v1/tasks 成功 |
| `task_planning` | status | 任务规划中 | AgentLoop 启动 |
| `agent_message` | dialog/status/debug | **通用消息**（见 §2.4） | 各阶段 |
| `agent_message_chunk` | dialog | 流式消息片段 | LLM 流式输出中 |
| `tool_started` | debug | 工具开始执行 | 工具调用前 |
| `tool_output` | debug | 工具执行输出 | 工具执行后 |
| `tool_finished` | debug | 工具执行完成 | 工具执行后 |
| `permission_required` | status | 需要用户授权 | 权限检查触发 |
| `permission_resolved` | status | 权限已裁决 | 用户批准/拒绝 |
| `file_changed` | status | 文件已变更 | 工具写入文件后 |
| `task_completed` | status | 任务完成 | AgentLoop 正常返回 |
| `task_failed` | status | 任务失败 | AgentLoop 异常返回 |
| `task_cancelled` | status | 任务已取消 | POST /cancel 调用 |
| `stream_end` | — | SSE 流结束 | 连接关闭时 |

> **注意**: `agent_message_chunk` 和 `stream_end` 不在 `EventType` 枚举中，由 `SSEGateway::pushStream()` 和 `streamTaskEventsImpl()` 直接构造。

### 2.4 metadata 规范（按频道约定）

metadata 是所有前端的消费路由依据。`channel` 字段决定消息应该显示在哪个区域。

#### 频道: `dialog` — 对话消息（用户可见的聊天内容）

```json
{
  "channel": "dialog",
  "expert": "summarizer",
  "streaming": false,
  "stream_end": true
}
```

| 字段 | 说明 |
|------|------|
| `channel` | 固定 `"dialog"` |
| `expert` | 产出消息的 Expert 名称 |
| `streaming` | 是否流式传输中 |
| `stream_end` | 流式传输是否结束（最终完整消息） |

**前端消费**: 渲染到聊天对话气泡中（assistant 侧）。

#### 频道: `status` — 任务状态（进度、阶段、路由切换）

```json
{
  "channel": "status",
  "expert": "executor",
  "stage": "expert_start",
  "next": "summarizer"
}
```

| 字段 | 可选值 | 说明 |
|------|--------|------|
| `stage` | `expert_start` / `expert_done` / `waiting` / `received` / `error` / `tool` | 当前阶段 |
| `expert` | Expert 名称 | 当前工作的 Expert |
| `next` | Expert 名称或 `_done` | 路由下一个目标 |
| `source` | `llm` / `tool` / `expert_loop` | 消息来源模块 |
| `rounds_left` | number | 当前 Expert 剩余轮次 |
| `size` | number | LLM 响应字符数 |
| `tool_name` | string | 被调用的工具名称 |
| `progress` | `{"current": N, "total": M}` | 进度信息 |

**前端消费**: 
- 渲染到状态面板 / 进度条
- `stage=expert_start` + `next` → 在流程图上高亮当前活跃节点和路由边
- `stage=expert_done` → 标记 Expert 节点完成
- `progress` → 更新进度条

#### 频道: `debug` — 调试信息（开发者面板）

```json
{
  "channel": "debug",
  "source": "llm_raw",
  "expert": "planner",
  "tool_name": "file.write"
}
```

**前端消费**: 渲染到调试/日志面板（默认折叠），普通用户不可见。

## 三、SSE 流生命周期

```
客户端                              服务端
  │                                   │
  │── GET /api/v1/tasks/:id/stream ──▶│
  │                                   │
  │◀── HTTP 200 + SSE headers ────────│  (连接建立)
  │                                   │
  │◀── event: task_created ───────────│  (历史回放开始)
  │◀── event: agent_message ──────────│
  │◀── event: task_completed ─────────│  (历史回放结束)
  │                                   │
  │◀── : ping\n\n ────────────────────│  (心跳，每 3 秒)
  │                                   │
  │◀── event: agent_message ──────────│  (实时事件)
  │◀── event: tool_started ───────────│
  │◀── event: tool_finished ──────────│
  │                                   │
  │◀── event: task_completed ─────────│  (终端事件)
  │◀── event: stream_end ────────────│  (流结束)
  │                                   │
  │── 连接关闭 ───────────────────────│
```

### 关键行为

1. **历史回放**: 连接建立后，服务端先推送该 task 已持久化的所有历史事件（从 SQLite 读取），再推送实时事件
2. **心跳**: 每 3 秒发送 `: ping\n\n`（SSE 注释行，客户端忽略）
3. **终端检测**: 除 SSE `task_completed/failed/cancelled` 事件外，每 9 秒轮询 DB 检查任务状态
4. **去重**: 客户端必须用 `id` 字段去重（历史回放 + 实时推送可能在边界重复）
5. **断线重连**: 客户端应在 `onerror` 后重新发起 SSE 连接，或降级为轮询

### 4.1 Web 前端（当前实现路径）

**文件位置**: `frontend/web/src/hooks/useAgentRuntime.ts`

**连接方式**: 浏览器原生 `EventSource`

```typescript
const source = new EventSource(`/api/v1/tasks/${taskId}/events`);

// 注册 13 种事件监听
const SSE_EVENT_NAMES = [
  "task_created", "task_planning", "agent_message",
  "tool_started", "tool_output", "tool_finished",
  "permission_required", "permission_resolved",
  "file_changed", "task_completed", "task_failed",
  "task_cancelled", "stream_end",
];

for (const eventName of SSE_EVENT_NAMES) {
  source.addEventListener(eventName, (message) => handleEvent(eventName, message));
}
```

**当前已适配**: ✅ 13 种事件类型全部监听，去重逻辑完备，降级轮询完备

**当前待适配**:

- ❌ 新事件类型（`agent_message_chunk` 流式片段）未被显式处理（`onmessage` 兜底）

### 4.2 C++ TUI 前端（同进程回调方式）

**文件位置**: `frontend/api-client/EventStreamClient.cpp`（HTTP SSE 方式，placeholder）
              `frontend/app/TuiApplication.cpp`（应用入口，placeholder）

**推荐方式**: 直接订阅 EventBus（零网络开销）

```cpp
// 在 TUI 应用初始化时订阅
auto& bus = ToolSystem::getInstance().eventBus();

bus.subscribeByTaskId(taskId, [&screen](const EventData& ev) {
    // 根据 metadata["channel"] 路由到不同面板
    std::string channel = ev.metadata.value("channel", "status");

    if (channel == "dialog") {
        // Post 到 FTXUI UI 线程更新对话框
        screen.Post([&chatView, ev]() {
            chatView.appendMessage(ev.content, "assistant");
        });
    } else if (channel == "status") {
        screen.Post([&statusPanel, ev]() {
            statusPanel.update(ev.content, ev.metadata);
        });
    } else if (channel == "debug") {
        screen.Post([&logPanel, ev]() {
            logPanel.append(ev.content);
        });
    }

    screen.Post(Event::Custom); // 触发重绘
});
```

## 六、协议兼容性检查清单

实现新前端时，必须验证以下所有场景：

- [ ] 任务创建 → 收到 `task_created` 事件
- [ ] LLM 等待 → 收到 `agent_message` + `stage:waiting`
- [ ] LLM 响应 → 收到 `agent_message` + `stage:received`
- [ ] 对话内容 → 收到 `channel:dialog` 的 `agent_message`
- [ ] 工具调用 → 收到 `tool_started` → `tool_finished` 序列
- [ ] 权限请求 → 收到 `permission_required`，批准后收到 `permission_resolved`
- [ ] Expert 切换 → 收到 `stage:expert_done` + `next:xxx`
- [ ] 任务完成 → 收到 `task_completed` + 最终 `agent_message`(channel:dialog)
- [ ] 流结束 → 收到 `stream_end`
- [ ] 断线重连 → 历史回放无重复
- [ ] 心跳 → 3 秒内收到 `: ping`
- [ ] 流式输出 → 收到多个 `agent_message_chunk` + 最终 `agent_message`(stream_end:true)

## 七、增量增强建议（不影响现有协议）

以下是纯增量、向后完全兼容的增强：

### 7.1 metadata 字段规范化（AgentLoop 侧）

当前 AgentLoop 的 metadata 已经相当规范，但部分字段使用 JSON 字符串硬编码。建议统一为结构化 JSON：

```cpp
// 当前（可用，但不易扩展）
SSEGateway::getInstance().pushStatus(
    taskId, "Expert [planner] 开始工作",
    EventType::AgentMessage,
    R"({"expert":")" + currentExpert->name + R"(","stage":"expert_start"})");

// 建议（结构化，便于前端解析）
json meta;
meta["channel"] = "status";  // 可由 pushStatus 自动注入
meta["expert"] = currentExpert->name;
meta["stage"] = "expert_start";
SSEGateway::getInstance().push(taskId, EventType::AgentMessage, content, meta,
                               SSEGateway::Channel::Status,
                               SSEGateway::Persist::Always);
```

> 注意：SSEGateway 的便捷别名（`pushStatus`/`pushDebug`）在内部会调用 `push()`，其 Channel 参数已由别名决定，但 metadata 仍需调用方手动构建。当前实现完全正确，增强仅为代码可读性。

### 7.2 Expert 节点高亮（前端侧）

web 前端可通过 metadata 实现流程图实时高亮：

```typescript
// 在 handleEvent 中
if (event.metadata?.channel === "status") {
  const expert = event.metadata.expert;
  const stage = event.metadata.stage;
  const next = event.metadata.next;

  if (stage === "expert_start") {
    highlightNode(expert, "active");
  }
  if (stage === "expert_done") {
    highlightNode(expert, "completed");
    if (next && next !== "_done") {
      highlightEdge(expert, next, "active");
    }
  }
}
```

### 7.3 三频道 UI 路由（前端侧）


## 八、常见问题

**Q: 为什么 SSE `event:` 行的类型名和 JSON `type` 字段值相同？**
A: 确保一致性。SSE `event:` 用于浏览器端 `addEventListener` 注册，JSON `type` 用于历史回放查询时的类型识别。两者必须一致。

**Q: 历史回放和实时推送如何去重？**
A: 使用 `id` 字段。服务端保证同一事件 ID 全局唯一。客户端维护 `seenEventIds` 集合（当前 web 前端已实现）。

**Q: C++ GUI 是否必须使用 HTTP SSE？**
A: 不必须。同进程可直连 EventBus，零开销。远程 GUI 建议使用 HTTP SSE 或 WebSocket。协议相同，仅传输层不同。

**Q: `agent_message_chunk` 何时推送？**
A: 当前 AgentLoop 暂未启用流式 LLM 输出（使用 `LlmClientFacade::chat()` 一次性返回）。启用流式后，通过 `SSEGateway::pushStream()` 推送 chunk。
