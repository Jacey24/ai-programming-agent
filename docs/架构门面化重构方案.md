## ⚠️ 重点关注（开发者在继续开发前务必阅读）

### 1. Agent Core 已大幅改造

`Agent.h` / `Agent.cpp` 不再持有 `sqlite3*` 指针，不再直接构造 `LogRepository` / `EventRepository` / `ToolCallRepository` / `FileChangeRepository`，不再直接调用 `LlmClient::chat()`。**所有操作通过门面单例完成。**

### 2. 新增门面基础设施（AppBootstrap 已初始化）

| 门面 | 初始化代码 | 状态 |
|------|----------|------|
| `DataAccessFacade` | `DataAccessFacade::getInstance().init(dbPath)` | ✅ 已就绪 |
| `SSEGateway` | `SSEGateway::getInstance().init(eventBus, dataFacade)` | ✅ 已就绪 |
| `LlmClientFacade` | `LlmClientFacade::getInstance().init("config/llm.json")` | ✅ 已就绪 |
| `ToolSystem` | `ToolSystem::getInstance().init(...)` | ✅ 已有 |

### 3. SSE 事件现在携带 channel 标识

每个 SSE 帧的 `metadata` 中新增 `channel` 字段：`"dialog"` / `"status"` / `"debug"`。前端目前的 `eventReceived` reducer **不区分 channel**，所有内容落入对话框。前端适配方案见 §6.2。

### 4. 配置文件变更

`config/agent.json` 已精简：LLM/security/agent 配置块被移除（改用 `config/llm.json` + `config/tools.json` + API 参数）。AppBootstrap 中的 `initializ_database()` 不再直接调用 Repository initTable——建表由 `DataAccessFacade::init()` 统一完成。

### 5. CMakeLists.txt 变更

`codepilot-agent-server` 目标新增 3 个源文件：`DataAccessFacade.cpp`、`LlmClientFacade.cpp`、`SSEGateway.cpp`。

## §3 配置清理 (v1.3)

### 3.1 `config/agent.json` 精简

该文件被 `AppBootstrap` 通过原始字符串解析读取 `storage.path` 和 `workspace.root`。以下配置块零代码引用，已移除：

| 移除项 | 替代方案 |
|--------|---------|
| `llm` 块 | `config/llm.json` + `LlmClientFacade` 多 provider |
| `security` 块 | `config/tools.json` 的 `risk_level` 驱动 |
| `agent.max_steps` / `tool_timeout_seconds` | `AgentConfig` 默认值 + API 参数 |
| `workspace.allow_outside_workspace` 等 | `config/workspace.json` 独立文件 |

保留：`storage`、`workspace.root`、`server` 三个有效配置块。

### 3.2 `AppBootstrap.cpp` 门面违规修复

`initialize_database()` 中移除了 `PermissionRepository(db).initTable()` 等 4 个直接 Repository 调用，建表职责统一由 `DataAccessFacade::init()` 内部完成。

## §5 封装方法论（周子涵五步分析法）

```
① 文档确认 → 查看架构文档，明确当前模块必须实现的功能
② 现状调研 → 查看当前代码中的 Service / Repository 实现
③ 冗余规划 → 列举可额外提供的便利性、统计性、管理性功能
④ 原子化   → 按理想设计将功能拆分为原子操作
⑤ 接口设计 → 规划门面类的对外接口（init + 原子方法 + 子系统引用）
```

### 核心原则

| 原则 | 说明 |
|------|------|
| **增量兼容** | 只增不减 — 新增门面类，保留原有 Service/Repository 内部不变 |
| **最终消费者最后搞** | Core Loop 和 API 层作为消费方，等所有服务提供者门面就绪后再封装 |
| **按依赖链排序** | 先做零依赖、被全部模块依赖的基础设施层 |
| **参照 ToolSystem** | 所有门面遵循相同的单例 + init + 子系统访问模式 |

## §6 SSE 三通道映射方案 (v1.4)

### 6.1 后端视图

| 通道 | `metadata.channel` | 语义 | Agent.cpp 调用方 |
|------|-------------------|------|-----------------|
| **dialog** | `"dialog"` | LLM 面向用户的自然语言对话 | LLM 回复非 DONE/FAIL 时 `pushDialog` |
| **status** | `"status"` | 任务生命周期事件 | `publishTaskEvent` → `pushStatus` |
| **debug** | `"debug"` | 主循环内部状态、Prompt、工具调用 | `publishDebugLog` 覆盖 5 个 source |

### 6.2 前端视图 ⚠️ 待适配

**当前行为**：`handleEvent` 不区分 channel，所有 `agent_message` 事件都进入对话框 reducer。

**预期适配**（前端开发参考，TypeScript）：

```typescript
// 当前：前端 useAgentRuntime.ts 第 230-242 行
const handleEvent = (eventName: string, message: MessageEvent<string>) => {
    const event = parseSseEvent(eventName, message);
    dispatch({ type: "eventReceived", event: { ...event } });
    // ← 所有内容进入同一 reducer
};

// 目标：按 channel 分流
const handleEvent = (eventName: string, message: MessageEvent<string>) => {
    const event = parseSseEvent(eventName, message);
    const channel = event.metadata?.channel || "status"; // 兼容旧数据
    switch (channel) {
        case "debug":
            dispatch({ type: "debugReceived", event });  // → 调试面板
            break;
        case "dialog":
            dispatch({ type: "dialogReceived", event }); // → 对话面板
            break;
        default:
            dispatch({ type: "eventReceived", event });  // → 状态栏（保持原逻辑）
    }
};
```

**协议约束**：`event:` 标签不变（13 种），`data` JSON 六字段不变，`metadata.channel` 为安全新增字段。前端不修改仍能正常运行（当前状态）。
